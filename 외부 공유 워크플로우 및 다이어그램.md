# 외부 공유 워크플로우 및 다이어그램

본 문서는 **Security Roadmap Agent** 시스템의 Vercel 프론트엔드(HTTPS)와 로컬 사설망 내의 AI 백엔드 API(Flask, Ollama, ChromaDB) 간 연동 구조, 보안 통제 우회, 그리고 LocalTunnel 타임아웃 극복을 위한 비동기 상태 조회의 데이터 흐름 및 상호작용 프로세스를 시각화합니다.

---

## 1. 물리적 네트워크 및 서비스 구조도

Vercel은 외부 사용자를 향한 게이트웨이 역할을 수행하며, 로컬 환경은 사설망에 위치하여 민감 정보를 처리합니다. 두 레이어는 LocalTunnel의 HTTPS 중계 터널링 채널을 매개로 소통합니다.

```mermaid
graph TD
    subgraph Public_Cloud ["외부 퍼블릭 클라우드 영역"]
        V[Vercel 정적 사이트<br/>https://security-loadmap-ax.vercel.app]
        LT_GW[LocalTunnel Gateway<br/>https://chilly-jokes-trade.loca.lt]
    end

    subgraph User_Browser ["사용자 브라우저 기기 (Client)"]
        Browser[웹 브라우저 Chrome/Safari]
    end

    subgraph Private_Network ["로컬 PC 사설 네트워크망 (Server)"]
        LT_Client[LocalTunnel Client<br/>npx localtunnel --port 5000]
        Flask[Flask REST API Server<br/>http://localhost:5000]

    end

    %% 연결 관계

    WSGI <-->|6. 단일 PNA 헤더 주입 및 정화| Flask
    Flask <-->|7. 시맨틱 RAG 컨텍스트| VDB
    Flask <-->|8. 구조화 JSON 추론| sLLM
    Flask -->|9. 동적 행 삽입/높이조절/스타일링| Excel
```

## 2. Chrome PNA (Private Network Access) 사전 승인 시퀀스

크롬 브라우저가 외부 퍼블릭 영역(Vercel)에서 사설 도메인(LocalTunnel 리다이렉트망)으로 접근 시, PNA 규격에 의거해 예비 요청(`Preflight request`)을 먼저 보낸 뒤 실제 데이터 요청을 전송하는 핸드셰이크 프로세스 다이어그램입니다.

```mermaid
sequenceDiagram
    autonumber
    actor User as 사용자 브라우저 (Chrome)
    participant Vercel as Vercel 프론트엔드
    participant GW as LocalTunnel 게이트웨이
    participant Middleware as WSGI PNAMiddleware
    participant Flask as Flask 백엔드 API

    Note over User, Vercel: 사용자가 Vercel 사이트에 접속하여 [연결하기] 또는 파일 업로드/분석 수행
    User->>GW: 1. [Preflight] OPTIONS /api/map (예비 요청 전송)
    Note right of User: 헤더 포함:<br/>Access-Control-Request-Private-Network: true

    GW->>Middleware: 2. 예비 요청 전달 (HTTP OPTIONS)
    Middleware->>Flask: 3. CORS 처리 요청
    Flask-->>Middleware: 4. 응답 생성 (헤더 중복 가능성 존재)
    
    Note over Middleware: [PNA 우회 핵심]<br/>헤더 목록에서 기존 PNA 헤더를 완전 필터링 제거 후<br/>Access-Control-Allow-Private-Network: 'true' 단일값으로 강제 신설
    
    Middleware-->>GW: 5. 정제된 HTTP 200 OK 응답 반환
    GW-->>User: 6. 예비 요청 통과 허가 (CORS PNA 인증 성공)

    User->>GW: 7. [Actual Request] POST /api/map (실제 파일 데이터 분석 요청)
    GW->>Middleware: 8. 실제 요청 전달
    Middleware->>Flask: 9. 백엔드 실 비즈니스 로직 수행 기동
    Flask-->>User: 10. HTTP 200 OK (작성 결과 데이터 반환 완료)
```

## 3. 비동기 분석 기동 및 1.5초 실시간 폴링 (Polling) 워크플로우

로컬서버-게이트웨이 간 60초 타임아웃(504 Gateway Timeout) 한계를 극복하고 대용량 AI 분석 진행률을 실시간 브라우저에 표시하기 위해 설계된 비동기 상태 추적 파이프라인의 구조입니다.

```mermaid
sequenceDiagram
    autonumber
    actor Browser as 브라우저 (Vercel UI)
    participant Flask as Flask API 백엔드
    participant Thread as 백그라운드 AI 스레드
    participant Ollama as Ollama sLLM (로컬 Gemma 2)

    Browser->>Flask: 1. POST /api/map (체크리스트 업로드 및 분석 요청)
    Note over Flask: 1. 고유 작업 번호 생성 (task_id)<br/>2. 스레드 풀 할당 및 AI 비동기 백그라운드 호출
    Flask-->>Browser: 2. HTTP 200 OK (즉시 task_id 반환하여 연결 조기 종결)
    
    Note over Browser: 진행도 0% 렌더링 및 1.5초 타이머 시작 (폴링 루프 진입)

    par AI 백그라운드 분석 진행
        Thread->>Ollama: 3. RAG 솔루션 Context 기반 Gemma 2 질의 (1번째 결함)
        Ollama-->>Thread: 4. 매핑 JSON 결과 반환
        Note over Thread: 분석 진척도 업데이트 (1/35 완료...)
    and 1.5초 주기 상태 폴링
        Browser->>Flask: 5. GET /api/map/status/{task_id} (1차 진행도 요청)
        Flask-->>Browser: 6. HTTP 200 OK {progress: 3, completed: 1, total: 35, status: 'running'}
        Note over Browser: UI 진행률 바 3% 업데이트
    end

    Note over Thread: 35개 전결함 매핑 분석 종료
    Thread->>Flask: 7. 작업 상태 세션 완료 처리 (status: 'completed', results: [...])

    Browser->>Flask: 8. GET /api/map/status/{task_id} (최종 폴링)
    Flask-->>Browser: 9. HTTP 200 OK {progress: 100, completed: 35, total: 35, status: 'completed'}
    Note over Browser: 1. 타이머 루프 해제 (100% 완료)<br/>2. 최종 매핑 테이블 렌더링 및 [최종 엑셀 로드맵 추출] 버튼 활성화
```

## 4. 엑셀 동적 병합 및 가시성 최적화 최종 추출 파이프라인
