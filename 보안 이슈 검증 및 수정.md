본 문서는 **정보보안감사 보안솔루션 로드맵 자동 수립 시스템**을 배포 및 운영하는 과정에서 직면했던 브라우저 보안 통제 및 컴플라이언스 규격 관련 주요 이슈들의 상세 원인과 이를 해결하기 위해 조치한 기술적 디버깅 결과들을 명세합니다.

## 1. Mixed Content (보안/비보안 혼합 콘텐츠) 제한

### 1.1 현상 및 원인
- **현상**: Vercel에 정식 배포된 웹 서비스(`https://security-loadmap-ax.vercel.app`)에서 사용자가 로컬 PC에서 띄운 백엔드 API 주소(`http://localhost:5000` 또는 `http://127.0.0.1:5000`)로 요청 시, 콘솔에 에러가 출력되며 비동기 통신이 전면 차단되었습니다.
- **원인**: 최신 브라우저는 보안 연결(**HTTPS**)로 서빙된 페이지 내에서 비암호화 연결(**HTTP**) 엔드포인트로 Ajax(Fetch, Axios) 요청을 보내는 것을 **Mixed Content(혼합 콘텐츠)** 취약점으로 간주하여 원천 차단하기 때문입니다.

### 1.2 해결 조치
- 로컬 백엔드 서버 앞에 외부 터널링 중계 게이트웨이(`localtunnel`)를 기동하여 암호화 인증서가 발급된 **HTTPS 공인 터널 도메인**을 실시간 바인딩했습니다.
- 프론트엔드 UI 연결 주소 제어창에 `https://{임시도메인}.loca.lt` 형식의 암호화 도메인을 지정하게 하여 혼합 콘텐츠 브라우저 검열을 우회했습니다.

## 2. Chrome PNA (Private Network Access) 차단 우회 및 CORS 중복 결함

### 2.1 PNA 보안 규격 개요
- 최신 크롬(Chrome 94+) 브라우저는 공인 네트워크(예: Vercel 도메인)의 정적 웹페이지가 사용자의 사설 네트워크 내부 자원(Localhost, 127.0.0.1, 192.168.x.x 등 로컬망 주소)으로 요청을 보낼 때, 중간자 공격이나 내부망 침투를 예방하기 위해 **Private Network Access(PNA)** 정책을 적용하여 접근을 규제합니다.
- 외부 도메인에서 로컬망 자원으로 API를 호출할 때는 예비 요청(`CORS Preflight`) 단계에서 반드시 아래의 PNA 허용 응답 헤더가 전송되어야만 접근이 승인됩니다.
  ```http
  Access-Control-Allow-Private-Network: true
  ```

### 2.2 중복 헤더 결함 및 WSGI 미들웨어 해결
- **결함**: Flask 백엔드에서 `@app.after_request` 데코레이터를 이용해 수동으로 PNA 헤더를 추가하거나 `flask-cors` 라이브러리를 동시 구동했을 때, 브라우저 콘솔에 PNA 헤더에 대한 CORS 거부 에러가 수시로 발생했습니다.
- **원인**: Flask의 응답 생성 과정에서 CORS 관련 헤더가 중복해서 가산되거나, `flask-cors`와 사용자 정의 훅이 충돌하여 `Access-Control-Allow-Private-Network: true, true` 또는 `true, false` 형태로 **다중 병합된 헤더값이 반환**되었기 때문입니다. 브라우저는 단일 `'true'` 외의 병합된 문자열을 비정상적인 응답으로 판단해 통신을 차단했습니다.
- **해결 조치**:
  - Flask 애플리케이션의 엔트리포인트(`app.py`) 단에서 CORS 헤더를 완전 정화하여 단일값으로 강제 주입해 줄 수 있는 WSGI 최상위 미들웨어인 **`PNAMiddleware` 클래스**를 설계 및 도입했습니다.
  ```python
  class PNAMiddleware(object):
      def __init__(self, wsgi_app):
          self.wsgi_app = wsgi_app

      def __call__(self, environ, start_response):
          def custom_start_response(status, headers, exc_info=None):
              # 기존 헤더 중 중복 처리된 Access-Control-Allow-Private-Network를 전부 필터링 제거
              new_headers = [h for h in headers if h[0].lower() != 'access-control-allow-private-network']
              # 오직 단일값 'true'로 깨끗하게 헤더를 신설
              new_headers.append(('Access-Control-Allow-Private-Network', 'true'))
              return start_response(status, new_headers, exc_info)
          return self.wsgi_app(environ, custom_start_response)

  # WSGI 미들웨어 등록
  app.wsgi_app = PNAMiddleware(app.wsgi_app)
  ```
  - 이 조치를 통해 어떤 서드파티 라이브러리 및 데코레이터와 얽히더라도 PNA 승인 헤더가 완벽히 1개(단일값 `true`)로만 깔끔하게 출력되어, 크롬 브라우저의 PNA 접근 통제 경고를 안전하게 완벽 우회했습니다.

## 3. 브라우저 세이프 브라우징 (Safe Browsing) 파일 다운로드 차단 극복

### 3.1 현상 및 원인
- **현상**: 매핑 결과를 엑셀로 저장하기 위해 프론트엔드에서 최종 다운로드 실행 시, 크롬 브라우저 우측 상단에 "안전하지 않은 파일이 차단됨" 또는 "보안 연결이 아니므로 다운로드할 수 없음"과 같은 붉은색 세이프 브라우징 경고 배지가 출력되며 저장 기능이 유실되었습니다.
- **원인**: JavaScript 비동기 `fetch(..., {method: 'POST'})`를 이용해 다운로드 응답을 바이트 스트림(Blob)으로 변환 후 `URL.createObjectURL(blob)`로 가상 메모리 링크를 생성하여 다운로드하는 기법은 크롬 브라우저의 최신 보안 필터망(Untrusted Content Origin)에 의해 파일 유출 시도로 분류되어 차단 대상이 되기 때문입니다.

### 3.2 해결 조치
- 정적 브라우저 내에서 가상의 임시 HTML `<form>` 엘리먼트를 동적으로 생성하여 엑셀 변환할 데이터 JSON 객체를 input 값에 직렬화해 담은 뒤, 백엔드 `/api/export` 엔드포인트로 직접 네이티브 **Form POST Submit**을 수행하게 클라이언트 코드(`app.js`)를 전면 개편했습니다.
- 백엔드(`/api/export`)에서도 전송 스트림 헤더에 브라우저 강제 형 검사 우회를 위한 보안 속성을 보강했습니다:
  ```python
  response.headers['X-Content-Type-Options'] = 'nosniff'
  response.headers['Cache-Control'] = 'no-cache, no-store, must-revalidate'
  response.headers['Pragma'] = 'no-cache'
  response.headers['Expires'] = '0'
  ```
- 이 폼 전송 방식 전환을 통해, 브라우저가 네이티브 다운로드 세션으로 안전하게 인식하여 경고 창 차단 없이 한 번에 엑셀 다운로드가 즉시 수행되도록 보안 조치를 성공시켰습니다.

## 4. Git 배포 단계에서의 계정 검증 및 SSH 보안 제한 결함

### 4.1 현상 및 원인
- **현상**: Vercel에 최신 코드를 배포하기 위해 리포지토리 원격지로 `git push` 실행 시, `Permission to GitHub repository denied` 에러와 함께 원격 커밋 전송이 전면 차단되었습니다.
- **원인**: 사용자의 Git 환경설정 상 원격 리포지토리의 소유자인 `jongcold` 계정(OAuth 키 또는 SSH 키 자격)과 로컬에 바인딩되어 통신을 시도하던 로컬 인증 키가 일치하지 않고, 설정된 글로벌 이메일 계정이 다른 협업 계정으로 인식되어 접근 통제 정책에 걸렸기 때문입니다.

### 4.2 해결 조치
- 로컬 리포지토리의 인증 정보가 들어있는 `.git/config` 파일의 `url` 정보와 원격 리포지토리 계정 매칭 수준을 점검했습니다.
- 로컬 저장소 수준에서 Git 사용자명 및 이메일을 소유자 계정(`jongcold`, `kjhkjh10114@gmail.com`)으로 명시적으로 수동 동기화하여 인증 충돌을 방지했습니다.
  ```bash
  git config user.name "jongcold"
  git config user.email "kjhkjh10114@gmail.com"
  ```
- 이후 충돌하는 깃 히스토리 정렬을 위해 리포지토리 강제 푸시(`git push -f origin main`)를 실행하여 Vercel 최신 빌드 배포를 정상 유도했습니다.
